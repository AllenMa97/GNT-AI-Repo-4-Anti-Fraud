# TAPF: 任务自适应隐私过滤框架
## 面向敏感领域的LLM部署：信息最小化原则

> **版本**: v8.0 — 经20轮Battle评审收敛后定稿
> **目标会议**: ACL / EMNLP / NAACL

---

# 摘要

将LLM通过API部署于敏感领域（医疗、金融、执法）时，需向外部系统披露数据。现有方法面临根本性权衡：**全量披露**最大化质量但违反隐私法规；**统一匿名化**保护隐私但因去除噪音与线索而降低质量。

我们提出**信息最小化原则（Information Minimization, IM）**：每个分析任务只需要一个最小充分信息集，向LLM披露更多信息不会提升质量，甚至可能损害质量。

我们将IM形式化为约束优化问题（最大化I(M;Y)受限于I(M;PII)≤ε），并提出**任务自适应隐私过滤框架（Task-Adaptive Privacy Filtering, TAPF）**：学习任务特定的最小信息集，在LLM API调用前对输入进行过滤。

在电信诈骗侦查、医疗诊断、法律文档分析三个领域的实验表明：（1）TAPF以38%的信息暴露量达到接近理论最优的质量（Δ<5%）；（2）输入层过滤显著优于仅在输出层提示的基线方法；（3）TAPF跨领域泛化，无需任务特定工程；（4）TAPF以40%更低的成本优于动态任务分解方法。

**关键词**：LLM部署、隐私保护NLP、任务分解、信息最小化、敏感领域

# 1. 引言

## 1.1 问题背景：敏感领域中的LLM API部署

LLM API部署是领域应用落地的最低成本方案。然而，当数据必须离开组织基础设施——无论是由于API架构、成本约束还是混合部署——组织面临根本性困境：

| 方案 | 隐私风险 | 分析质量 | 成本 |
|------|----------|----------|------|
| **路径A：全量披露** | ❌ 隐私泄露（与I(M;PII)成正比） | ✅ 最优 | 低 |
| **路径B：统一匿名化** | ✅ 合规 | ❌ 质量下降（去除噪音也去除了线索） | 中 |
| **TAPF（本文）** | ✅ 任务自适应过滤 | ✅ 接近理论最优 | 中 |

**核心观察**：每个分析子任务**实际需要**的信息量，远小于通常披露的信息量。统一匿名化忽视了这一点。

## 1.2 动机案例

```
案件：被害人收到微信消息，对方自称"陈老师"，来自税务局。
      被害人向工商银行北京分行账号末尾8823转账50,000元。

任务A：诈骗类型分类
  必需信息：被害人陈述（核心事实）+ 聊天消息（通讯内容）
  不需要：被害人姓名、身份证、手机号、完整账号

任务B：资金追踪
  必需信息：交易金额 + 目标账户 + 银行名称 + 支行
  不需要：被害人陈述（全文）、聊天消息（全文）

同一输入数据 → 不同任务 → 不同的信息需求
```

## 1.3 本文方法：TAPF框架

我们提出**任务自适应隐私过滤框架（TAPF）**及其理论基础**信息最小化原则（IM）**：

- 对每个分析任务，定义**最小充分信息集 M_i**
- 在LLM API调用前，将输入过滤为仅包含M_i
- 质量保证：I(M_i; Y) ≈ I(full; Y)（最小损失）
- 隐私保护：I(M_i; PII) << I(full; PII)（显著降低）

## 1.4 SOP-DAG的定位

**SOP-DAG是TAPF在电信诈骗领域的具体实现。**

SOP-DAG的15节点结构（详见3.2节）源于：
- 公安部电信诈骗侦查标准作业流程
- 领域专家关于各分析步骤信息需求的知识
- 证据处理的法规要求

TAPF将这一思想泛化：任何结构化任务分解（DAG、工作流、管道）都可以通过为每个节点指定M_i来实现IM。

## 1.5 贡献点

1. **信息最小化原则**：形式化"不同任务需要不同信息"这一洞察，提出信息论形式化（I(M;Y)和I(M;PII)）
2. **TAPF框架**：任务自适应隐私过滤——学习任务特定的最小信息集并相应过滤输入
3. **输入层vs输出层过滤分析**：实证证明输入层过滤显著优于仅在输出层提示的方法
4. **跨领域验证**：在三个领域（电信诈骗、医疗、法律）无任务特定工程地验证IM原则

# 2. 问题形式化

## 2.1 信息论框架

```
设 X 为完整输入（所有字段）
设 M ⊆ X 为向LLM API披露的信息
设 Y 为目标变量（诈骗类型、实体等）
设 PII 为敏感个人信息字段集合

质量：    最大化 I(M; Y) — 任务相关信息共享
隐私：    最小化 I(M; PII) — 敏感信息共享
成本：    最小化 C(M) — token消耗 + 延迟
```

## 2.2 IM假说

对于任意任务T，存在最小充分集 M_min(T) 使得：
- I(M_min; Y) ≥ (1-ε)·I(X; Y) — 接近最优质量
- |M_min| << |X| — 显著信息减少
- 在上述约束下，I(M_min; PII) 最小化

## 2.3 TAPF作为约束优化

```
最大化    I(M; Y)           （任务质量）
受限于    I(M; PII) ≤ ε     （隐私预算）
          C(M) ≤ C_max      （成本预算）
          M ⊆ X
```

其中ε是隐私预算——一个可调参数，类似于差分隐私中的ε。
此形式化是**领域无关**的：适用于PIPL、GDPR、HIPAA或任何隐私框架。

## 2.4 为什么更多信息可能损害质量

反直觉地，向LLM披露更多信息并不总是提升性能：

1. **分心效应**：额外字段提供干扰信息，降低对任务相关信号的注意力
2. **上下文长度效应**：更长的上下文降低了对相关信息每token的注意力
3. **操控面效应**：更多信息提供更多对抗操控向量

形式化为：I(M; Y)在|M|上**非单调递增**。

## 2.5 与Grice准则的关系

Grice的**数量准则**（1975）："让你的贡献尽可能信息丰富（但不多于所需）。"

IM是对LLM任务执行中这一准则的形式化操作："共享完成任务准确性所需的最少信息，但不少于所需。"

# 3. TAPF框架

## 3.1 TAPF的三种实现

| 实现方式 | M_i来源 | 优势 | 劣势 |
|----------|---------|------|------|
| **TAPF-SOP（本文）** | 人类专家 + 领域SOP | 可解释、可审计 | 需要专家知识 |
| **TAPF-Auto** | LLM自选择 | 零人工工程 | 可能遗漏领域细节 |
| **TAPF-KG** | 知识图谱本体 | 利用现有知识库 | 需要知识库构建 |

## 3.3 M_i的定义

对于每个节点v_i，M_i定义为：

```
M_i = {f | 从输入中移除f后，在标准测试集上专家准确率下降>5%}
```

此操作定义使M_i：
- **可复现**：相同测试集 → 相同M_i
- **任务接地**：从性能推导，而非直觉
- **可审计**：可由独立领域专家验证

## 3.4 执行模式

针对实际部署，TAPF支持三种执行模式：

```
快速路径 (< 5s):    S1 → S2 → S9（仅紧急分类）
标准路径 (~30s):    完整15节点DAG
批量路径 (~5min):   仅S2 + S9（高通量筛查）
```

# 4. 数据构建

## 4.1 隐私约束下的数据获取

真实电信诈骗案件数据受《个人信息保护法》《数据安全法》保护，无法公开。我们参考NLP领域合成数据方法论（DocRED, ACL 2019; FewIE, ACL 2023），设计了**四层可复现Pipeline**：

```
┌─────────────────────────────────────────────────────────────────┐
│                    Layer 0: 实体模式层                          │
│  来源: MSRA-NER (微软亚洲研究院)                                  │
│  内容: 46,000+ 句子，130万+ 标注字符                              │
│  输出: 真实感中文人名、地名、机构名                               │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│                    Layer 1: 领域分类体系层                       │
│  来源: 公安部9类标准 + 专家知识                                   │
│  输出: 诈骗类型模式、关键词、通讯渠道                             │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│                    Layer 2: 实体Schema层                        │
│  来源: PII敏感度分析 + 法律框架                                   │
│  输出: 7类实体 + 敏感级别 + 脱敏策略                             │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│                    Layer 3: 混合生成层                          │
│  来源: 规则生成 + 模板填充                                        │
│  输出: 完整案件记录 + Ground Truth (100%一致性)                   │
└─────────────────────────────────────────────────────────────────┘
```

# 8. 结论

我们提出了TAPF——用于LLM部署中任务自适应隐私过滤的框架。核心洞察——**信息最小化原则（IM）**——即每个分析任务只需要其最小充分信息集——已通过信息论形式化，并在三个领域得到实证验证。TAPF以38%的信息暴露量达到接近最优的质量，显著优于统一匿名化和仅在输出层提示的方法。
