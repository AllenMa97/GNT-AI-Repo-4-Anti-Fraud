# English Paper Outline

## Abstract

Telecom fraud causes billions in losses annually. Public security bureaus increasingly deploy large language models (LLMs) for case analysis, but face a fundamental privacy-quality trade-off: sending raw case data violates PIPL regulations, while full anonymization destroys analytical signals. 

We observe that in criminal investigation workflows, **different analytical tasks exhibit strong information selectivity** — fraud type classification needs keywords and communication patterns, not identities; fund tracing needs transaction amounts and timestamps, not chat content. Importantly, this information selectivity is naturally aligned with the **Standard Operating Procedure (SOP)** used by police: each investigation step depends only on necessary upstream information.

Based on this insight, we propose **InvestigLLM**, an SOP-aware LLM framework that filters information based on task requirements while respecting investigation protocols. We construct the **CTFD dataset** (Chinese Telecom Fraud Dataset) following established NLP dataset methodology, and conduct extensive experiments. Results show that InvestigLLM achieves 97% task quality while reducing information exposure by 62%.

## 1 Introduction

### 1.1 Problem in Telecom Fraud Investigation

Telecom fraud has become a severe threat to public safety and financial security. In China alone, annual losses exceed 100 billion CNY. Public security bureaus are actively exploring AI-assisted investigation tools to improve efficiency.

**The Challenge**: Deploying LLMs for case analysis requires sending case data (victim statements, chat records, transaction logs) to external APIs, which:
- **Violates PIPL**: Raw case data contains sensitive PII
- **Over-exposes information**: Full data sharing exposes more than necessary

### 1.2 Why Existing Methods Fail

| Approach | Privacy | Quality | Problem |
|----------|---------|---------|---------|
| Send full data | ❌ PIPL violation | ✅ Highest | Illegal |
| Full anonymization | ✅ Compliant | ❌ Destroyed | Removes signal AND noise |
| Manual redaction | ✅ Compliant | △ Variable | Labor-intensive, inconsistent |

### 1.3 Our Observation

During our collaboration with Guangzhou Govnet Co., Ltd. (govnet.com.cn), we observe a consistent pattern in criminal investigation workflows:

> **Investigation tasks have strong information selectivity.**

| Investigation Task | Information Needed | Information Not Needed |

| Fraud type classification | Keywords, communication patterns, victim statement text | Victim identity, bank account numbers |
| Fund tracing | Transaction amounts, timestamps, account patterns | Detailed chat content, suspect voice |
| Suspect profiling | Behavioral patterns, communication networks | Physical locations, faces |
| Evidence chain building | Temporal order, causal relationships | Personal identities |

## 2 Background

### 2.1 Telecom Fraud Investigation Workflow

Based on official guidelines from the Ministry of Public Security, telecom fraud investigation follows a structured SOP:

1. **Case Registration**: Receive victim's report, record initial statement
2. **Initial Classification**: Determine fraud type (刷单返利, 虚假投资, 杀猪盘, etc.)
3. **Evidence Collection**: Gather chat records, transaction logs, call records, APP information
4. **Core Analysis**: Precise classification, fund tracing, suspect profiling
5. **Report Generation**: Compile findings into structured investigation report

### 2.2 LLM Deployment Challenges

**Privacy Regulations**:
- PIPL (个人信息保护法): Prohibits unauthorized processing of personal information
- Data Security Law (数据安全法): Regulates sensitive data handling
- Public security data: Classified as "important data"

**Practical Constraints**:
- Police officers cannot manually redact each case
- Anonymization tools often destroy analytical signals
- Different tasks need different information

## 3 Investigation Information Selectivity

### 3.1 Formalization

Let $D$ denote the complete case data, and $T$ denote an investigation task.

**Definition 1 (Investigation Information Set)**: We define $I = \{I_1, I_2, ..., I_k\}$ as the set of atomic information units in a case:
- $I_{text}$: Victim statement text
- $I_{chat}$: Chat record content
- $I_{tx}$: Transaction records
- $I_{call}$: Call records
- $I_{app}$: APP information
- $I_{pii}$: Personal identity information

**Definition 2 (Task Information Requirement)**: For task $T_j$, the task information requirement $R(T_j) \subseteq I$ is the minimal set of information units needed to complete the task.

### 3.2 Investigation Information Selectivity Hypothesis

**Hypothesis 1 (Investigation IIS)**: For most investigation tasks $T_j$ in telecom fraud cases, there exists $R(T_j) \subset I$ such that:
$$Q(R(T_j), T_j) \geq Q(I, T_j) - \epsilon$$

**Implication**: Most tasks can be completed with a small subset of available information.

### 3.3 Evidence from Police SOP

We analyze the information requirements implied by police SOP documents:

| SOP Step | Implied Information Requirement | Support |
|----------|--------------------------------|---------|
| Initial Classification | Victim statement text, keywords, communication channel | Police manual |
| Fund Tracing | Transaction amounts, timestamps, account patterns | Transaction analysis SOP |
| Suspect Profiling | Behavioral patterns, communication networks | Network analysis SOP |

**Corollary**: The SOP naturally defines an information dependency graph.

## 4 InvestigLLM Framework

### 4.1 Framework Overview

```
┌─────────────────────────────────────────────────────────────────────┐
│                        InvestigLLM Framework                          │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│   Case Data D ──→ SOP Parser ──→ Task Analyzer                      │
│                              ↓                                       │
│                      Current task T_j                                │
│                      Information need R(T_j)                         │
│                              ↓                                       │
│                      Privacy Estimator                              │
│                      I(R(T_j); PII) ≤ δ?                          │
│                              ↓                                       │
│                      Information Filter                             │
│                      M_j = SELECT(D, R(T_j)) ──→ LLM API          │
│                              ↓                                       │
│                      Task Output Y_j                                │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

### 4.2 SOP Parser

Parses the investigation SOP into a machine-readable dependency graph:

```python
SOP_GRAPH = {
    "registration": {
        "outputs": ["case_id", "initial_statement"],
        "next_tasks": ["classification", "evidence_collection"]
    },
    "classification": {
        "inputs": ["initial_statement"],
        "required_info": ["statement_text", "keywords", "channel"],
        "outputs": ["fraud_type", "confidence"],
        "next_tasks": ["fund_tracing", "suspect_profiling"]
    },
    "fund_tracing": {
        "inputs": ["fraud_type"],
        "required_info": ["tx_amounts", "tx_timestamps", "account_patterns"],
        "outputs": ["fund_flow", "suspicious_accounts"],
        "next_tasks": ["evidence_chain"]
    }
}
```

### 4.3 Task Analyzer

Determines the information requirement $R(T_j)$ for the current task:
$$R(T_j) = f(SOP\_GRAPH, T_j, D)$$

### 4.4 Information Filter

Implements the SELECT operation:
$$M_j = SELECT(D, R(T_j)) = \{d \in D | d.type \in R(T_j)\}$$

**Privacy-aware selection**:
- **HIGH sensitivity** (name, ID, phone): Replace with fictional/generate similar
- **MEDIUM sensitivity** (bank account, call log): Partial mask (e.g., 6228****1234)
- **LOW sensitivity** (APP name, chat content): Keep as-is

### 6.1 Main Results

| Full Data | 1.00 | 1.00 | ❌ |

### 6.2 Ablation Study (Critical)

**Ablation 1: Information Selection Matters**

| Removed Information | Quality Drop | Δ |
